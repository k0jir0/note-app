DEVELOPMENT NOTES & LESSONS LEARNED
Note-Taking Application Development Process
================================================================================

PROJECT OVERVIEW

This full-stack note-taking application demonstrates the complete lifecycle of a production-ready web application, built with Node.js, Express, MongoDB, and EJS. The project features user authentication, RESTful CRUD operations, comprehensive input validation, and 122 passing tests. Through the development process, multiple challenges emerged that required careful consideration of security, testing strategies, and deployment readiness.

Recent updates (March 5, 2026) hardened validation and authentication flows:
- Added strict auth payload sanitization (`sanitizeAuthPayload`) and login password validation (`validateLoginPassword`).
- Restricted note API payloads to allowed fields only (`title`, `content`, `image`) with explicit rejection of unexpected fields.
- Trimmed and validated note route IDs before database operations in API and page routes.
- Fixed a Google-auth edge case where local login attempted `bcrypt.compare` against missing password hashes.
- Implemented fallback behavior: Google-only accounts are redirected to `/auth/login/federated/google`; other failures return a frontend login error.
- Added regression tests covering the Google-only login redirect path and new validation behavior.


MAJOR CHALLENGES AND SOLUTIONS

Authentication and authorization proved most challenging. Securing user sessions with Passport.js involved local strategy configuration, user data serialization/deserialization, and route protection middleware. The solution used reusable `requireAuth` middleware and user-specific queries to ensure users could only access their own notes. Separating API routes (JSON) from page routes (HTML) provided better error handling for different client types.

Input validation and sanitization balanced security with user experience. The challenge was preventing XSS attacks while validating diverse field types without exposing system internals. This was addressed by creating centralized validation utilities in `src/utils/validation.js`, implementing multi-layer validation across client, server, and database, and using an `isUpdate` flag to differentiate create from update operations. Inputs were sanitized using regex to remove dangerous script tags while preserving legitimate content.

Comprehensive test coverage without a live database connection demanded significant effort. Mocking Mongoose model methods and stubbing authentication middleware while ensuring test isolation was complex. Sinon for stubs, spies, and mocks alongside consistent test patterns solved this. Proper cleanup in `afterEach` blocks prevented state leakage, achieving 122 tests covering models, controllers, routes, and utilities.

Production deployment preparation revealed technical debt: hardcoded values, insecure cookie settings, and mixed indentation. Resolution involved environment variables with fallbacks, secure cookies for production (`secure: process.env.NODE_ENV === 'production'`), proper npm scripts, protected seed routes, and ESLint `--fix` for standardized formatting.

Error handling consistency required a centralized `handleError` utility standardizing response formats as `{ success, message, errors }` with appropriate HTTP status codes. All async handlers were wrapped in try-catch blocks, with dedicated test cases verifying database failures, validation errors, and invalid ObjectId casting.

Database schema design involved decisions about references versus embedded documents, field constraints, and indexing. ObjectId references for the User-to-Notes relationship with automatic timestamps, optional fields with defaults, and realistic constraints (title: 3-200 characters, content: max 10,000) created a flexible yet robust structure. Compound queries combining user ID and note ID ensured proper authorization.

The project initially pointed to a different repository and required migration. This involved removing the incorrect remote, installing GitHub CLI (`winget install GitHub.cli`), authenticating via web, creating the new repository programmatically (`gh repo create test-app --public --source=.`), and pushing the code successfully.

Legacy code maintenance required updating older controllers and routes that lacked authentication. Rather than deleting these files, they were updated with `requireAuth` middleware and user-specific filtering to maintain backward compatibility while improving security. Tests were updated accordingly with documentation explaining preservation.

Model naming standardization involved renaming `notes.js` to `Notes.js` for consistency with JavaScript conventions. This required updating all import statements across controllers (`noteController.js`, `noteApiController.js`), routes (`notePageRoutes.js`), test files, and `index.js`. Git's `mv` command preserved file history during the rename.


KEY LEARNINGS

Security must be foundational. Never trust user input—validate and sanitize everything. Hash passwords, use environment variables for secrets, implement authentication and authorization, configure cookies securely in production, and protect destructive operations.

Separation of concerns improves maintainability. Models handle data structure and validation, controllers contain business logic, routes handle HTTP mapping, middleware provides reusable functionality, utilities centralize common operations, and views contain presentation logic only.

Testing is non-negotiable. Write tests before or during development, not after. Mock external dependencies like databases and authentication, test success and error scenarios, focus on critical paths while aiming for high coverage, use descriptive test names, and clean up after each test.

Environment configuration requires discipline. Use .env files for local development but never commit secrets. Provide .env.example as a template, use different configurations for dev/test/prod, default to safe values, and document all required variables.

API design should follow consistent patterns. Use standardized response formats, RESTful conventions, appropriate HTTP status codes, detailed error messages for developers, partial updates with optional fields, and timestamps in all database records.

Error handling must be comprehensive and centralized. Use try-catch for async operations, centralize error logic, log errors for debugging but not in production, return user-friendly messages, differentiate client (4xx) from server errors (5xx), and never expose stack traces.

Code organization affects maintainability. Group related files logically, use consistent naming conventions, keep files focused on single responsibilities, comment complex logic but not obvious code, extract repeated code into utilities, and follow established patterns.

Documentation enables collaboration and future development. Write clear READMEs with setup instructions, document API endpoints with examples, include request/response formats, explain validation rules, provide troubleshooting guidance, and keep documentation synchronized with code.

Honest technical debt assessment beats false perfection. Grade realistically, identify high-priority issues versus nice-to-haves, fix production blockers before optimization, document known issues, and balance perfection with shipping functional software.

Git workflow discipline prevents problems. Commit frequently with descriptive messages, review changes before committing, use branches appropriately, keep commits focused, push regularly, and use version tags for releases.


TECHNICAL INSIGHTS

Mongoose quirks: validation runs on save but not update unless `runValidators: true` is specified, `findOne` returns null when no document is found, ObjectId casting errors need explicit handling, virtual fields don't persist, and timestamps only update on save/update operations.

Express sessions store data server-side while cookies hold only session IDs. Session secrets should be long and random, sessions persist across restarts when using a store (memory by default), Passport serializes only the user ID, and logout requires both `req.logout()` and session destruction.

Passport.js patterns: local strategy handles username/password authentication, serialization stores user identifiers in sessions, deserialization fetches the full user from the database, callbacks use `done(err, user)` or `done(err, false, { message })`, and `req.isAuthenticated()` checks login status. Local auth must guard against accounts without local hashes (for example Google-only users) before calling `bcrypt.compare`.

Testing best practices: use `describe()` for grouping tests, `beforeEach()` for setup, `afterEach()` for cleanup and stub restoration, stub database calls rather than connecting to real databases, test edge cases like empty strings and long inputs, and verify both success and error paths.

ESLint automation with `--fix` auto-corrects many issues. Configure rules in .eslintrc.json, accept some warnings as reasonable (unused variables in stubs), recognize consistent indentation improves readability, and run the linter before committing.


WHAT I WOULD DO DIFFERENTLY

Starting with test-driven development for critical features would have caught bugs earlier and established test patterns from the beginning. Planning authentication first—designing the auth flow before other features and creating the middleware library early—would have prevented security gaps.

Standardizing early by choosing indentation style, setting up linter configuration, and establishing naming conventions would have avoided cleanup work. Documenting during development rather than deferring keeps documentation accurate and complete.

Using environment variables from day one prevents refactoring hardcoded values later. Setting up .env.example immediately and configuring for production from the start saves time and reduces deployment issues.

Centralizing validation early by creating utilities before building features prevents scattered logic and encourages reuse. Implementing proper logging with Winston or Morgan, logging to files in production, and including request IDs would improve debugging.


FINAL REFLECTION

This project achieved 122 passing tests, zero ESLint errors, comprehensive authentication and authorization, input validation and sanitization, RESTful API design with proper status codes, production-ready configuration, a concise README, proper Git repository setup, logical code organization, consistent model naming (Notes.js), and security best practices.

Building software is inherently iterative. The first version may work, but achieving security, maintainability, and production readiness requires conscious effort and continuous improvement. The most valuable lesson is that honesty about technical debt surpasses claiming perfection. Real projects involve trade-offs, and understanding them makes for better development decisions.
