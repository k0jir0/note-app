PROJECT OVERVIEW

This full-stack note-taking, security-research, and browser-automation-aware application demonstrates the complete lifecycle of a production-ready web application, built with Node.js, Express, MongoDB, and EJS. The project features user authentication, RESTful CRUD operations, comprehensive input validation, dedicated Security Operations, Alert Triage ML, Mission Access Assurance, Hardware-Backed MFA, Session Security, Query Injection Prevention, XSS and CSP Defense, Server Access Control, Playwright Testing, Selenium Testing, and Self-Healing Locator Repair modules, and 575 passing tests. Through the development process, multiple challenges emerged that required careful consideration of security, testing strategies, deployment readiness, explainability, and the interaction between secure defaults and developer ergonomics.

Current verified state (March 31, 2026): 575 passing tests, ESLint clean, `npm audit` clean for both all-dependency and production-only gates, Chromium Playwright smoke coverage at 21 passing browser tests, MongoDB-backed session persistence, synchronizer-token CSRF protection, Helmet/CSP browser hardening, rate-limited security analysis endpoints, a unified Research Workspace, dedicated Security Operations, Alert Triage ML, Mission Access Assurance, Hardware-Backed MFA, Session Security, Query Injection Prevention, XSS and CSP Defense, Server Access Control, Playwright Testing, Selenium Testing, Self-Healing Locator Repair, Supply Chain Security, and Audit Trail and Telemetry modules, plus focused March 29 module coverage at 13 passing tests for the new page routes and persisted audit-history backend, focused March 30 sandbox runtime coverage at 20 passing tests, and focused March 31 hardening regressions at 82 passing tests. The app now enforces a global protected-API access gate, request-level Mongo injection blocking, strict CSP-backed rendering posture, mission-aware RBAC-plus-ABAC evaluation, hardware-first session assurance with WebAuthn and PKI support, strict idle and absolute session limits with concurrent-login invalidation, startup-stable local launch behavior, TLS 1.3-only inbound transport when HTTPS is enabled, sanitized metadata/error exposure, immutable remote logging, request-scoped database state-change telemetry, SIEM-friendly JSON or syslog log formatting, a validated hardened distroless container build, a verified three-zone DMZ/app/data sandbox deployment with an unprivileged Nginx edge tier, a live authenticated audit-history feed backed by persisted `AuditEvent` records, authenticated same-origin note-image delivery backed by server-managed assets, a reusable app factory and centralized route registry, persisted break-glass runtime state, account-level login lockout, privileged dev-tool gating, strict immutable-audit fail-closed behavior, fail-safe break-glass persistence degradation, federated-account lockout parity, and stricter MongoDB TLS plus immutable-audit enforcement across protected runtime deployments.

TROUBLESHOOTING (as of March 28, 2026)

- Semgrep rule schema errors: Add `languages: [javascript]` to `.semgrep.yml`.
- `npm ci` lockfile mismatch: Run `npm install`, commit the lockfile, retry.
- Node engine mismatch in CI: Update workflow to match required Node version.
- Optional dependency issues: Install only when needed, rely on lazy-require.
- Trivy scan failures: Artifacts are report-only; triage and patch as needed.
- Google OAuth/local auth issues: Ensure all OAuth env vars and redirect URIs match, use `.env.local` for workstation secrets.
- MongoDB/Redis startup: Wait for successful DB connection before listening; Redis is now optional and fails fast if not configured.



KEY LEARNINGS

- Security must be foundational: validate/sanitize all input, hash secrets, use environment variables, and enforce strict session/auth controls.
- Separation of concerns and modularity are critical for maintainability.
- Testing is essential—mock dependencies, cover edge cases, and clean up after each test.
- Environment config discipline prevents future refactoring and deployment issues.
- Consistent API design and error handling improve developer and user experience.
- Honest technical debt assessment and documentation are more valuable than chasing false perfection.
- Git workflow discipline and clear documentation enable collaboration and future development.



TECHNICAL INSIGHTS

- Mongoose: Always use `runValidators: true` on updates, handle ObjectId casting, and understand virtual/timestamp behavior.
- Express/Passport: Sessions are server-side, cookies hold only IDs, and logout requires both session destruction and `req.logout()`.
- CSRF: Synchronizer-token pattern works, but templates must be tested for token presence.
- CSP: Removing inline scripts enables strict policies.
- Testing: Use stubs/mocks, group/cleanup tests, and verify both success and error paths.
- ESLint: Automate with `--fix`, accept some warnings, and enforce consistent style.



WHAT I WOULD DO DIFFERENTLY

- Start with TDD for critical features and authentication.
- Standardize code style, linter config, and naming conventions from the beginning.
- Use environment variables and `.env.example` from day one.
- Centralize validation and logging early.
- Document as you go, not after the fact.
