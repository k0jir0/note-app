Title: Security Module — Design, Implementation, and Evaluation
Authors: Ryan Watkins
Date: March 13, 2026

Abstract
This paper presents the design, implementation, and empirical evaluation of the Security Module: a modular, extensible subsystem for detection, correlation, and investigative support within full-stack web applications. The Security Module integrates lightweight host- and application-level event ingestion, rule-based and heuristic alerting, and a correlation engine that groups related artifacts by semantic targets. We describe the architecture, core heuristics, data models, and evaluation methodology. Experimental results using synthetic and real-world workloads demonstrate that grouping correlations by meaningful entities (targets) reduces false-positive inflation, improves analyst signal-to-noise, and scales to realistic deployment sizes while maintaining low latency. We conclude with a discussion of limitations and directions for future research.

1. Introduction
Modern web applications face increasing volumes of security-relevant telemetry: host logs, network scan results, IDS alerts, and application-level events. Without careful aggregation and contextualization, this telemetry can overwhelm analysts and automated systems, producing high counts of low-value matches and obscuring true threats. This paper introduces the Security Module, a practical subsystem designed to integrate with existing Node.js + Express applications and MongoDB-backed persistence layers to provide: (1) event ingestion and normalization; (2) alerting heuristics; (3) correlation across heterogeneous sources grouped by semantically meaningful targets; and (4) concise, actionable summaries for analysts.

The paper makes three contributions:
- A modular architecture and data model for integrating heterogeneous security telemetry into full-stack applications.
- A correlation algorithm that shifts from pairwise scan×alert matching to grouping by semantic targets, reducing inflated correlation counts and improving signal clarity.
- An empirical evaluation showing the benefits of grouping and demonstrating the module's performance and usability characteristics.

2. Background and Motivation
Security instrumentation in web applications often produces many-to-many relationships between event sources (e.g., scans, alerts). Naive pairwise cross-product correlation can produce combinatorial inflation: when N scans and M alerts match on overlapping criteria, a pairwise approach yields up to N×M correlation items, many of which are redundant from an analyst's perspective. Prior work in SIEMs and security analytics advocates entity-centric correlation (e.g., host-centric, account-centric) to reduce noise and present a coherent investigative surface [1,2]. We build on these principles to design a correlation engine suitable for repository-hosted research and small-to-medium operational deployments.

3. System Design
3.1 Goals and Constraints
The Security Module is designed with the following goals:
- Integrability: minimal intrusion into existing Express-based apps.
- Extensibility: pluggable data normalization and heuristic rules.
- Analyst-focused outputs: group correlated items into representative entities (targets) with concise overviews.
- Performance: maintain low-latency responses for UI-driven workflows.

Constraints include deployment on commodity infrastructure and storage in MongoDB (document model). The design favors simplicity and explainability—critical for security teams that must trust automated correlations.

3.2 Architecture Overview
The module has four primary components:
- Ingest layer: collects events (alerts, scans, logs) from application and external sources. Events are normalized to a shared schema.
- Alerting engine: applies deterministic and heuristic rules to generate alert objects from normalized events.
- Correlation engine: computes relationships between alerts and scans, grouping related items by semantic `target` (e.g., IP address, host identifier, username).
- API and UI adapter: serves paginated, summarized data for analyst UIs and automation endpoints (sample injection, export).

3.3 Data Model
We represent core entities as JSON documents with the following minimal schema:
- Scan: { _id, target, source, timestamp, metadata }
- Alert: { _id, target, type, summary, timestamp, metadata }
- Correlation: { target, alertIds, scanIds, overview }

The `target` field is the canonical grouping key (e.g., an IP string or account identifier). Overviews contain computed metrics (counts, top types, recency) to drive UI cards.

4. Correlation Algorithm
4.1 Problem Statement
Given sets S (scans) and A (alerts), produce a compact set of correlation objects that represent related activity without producing redundant pairwise duplicates.

4.2 Naive Pairwise Approach
A baseline approach enumerates all s ∈ S and a ∈ A and emits a correlation for every matching pair. This generates up to |S|×|A| outputs, which is undesirable in noisy environments.

4.3 Group-by-Target Approach
We propose: (1) compute pairwise matches to identify candidate connections; (2) group candidate matches by a canonical `target` key; (3) merge group members into a single representative correlation with aggregated metadata and a concise overview. Algorithmically:
- For each scan s and alert a, if match(s,a) then add (s,a,target)
- Build a map M: target → {scanIds, alertIds, counts, examples}
- For each target in M produce a correlation object containing aggregated sets and overview stats

This reduces output size to at most the number of distinct targets, addressing the inflation problem and aligning with analyst mental models (investigate per host or account).

4.4 Heuristics and Matching
Matching is rule-driven. Example rules include exact target equality, CIDR containment for IP ranges, reverse-DNS overlaps, and fuzzy string similarity on hostnames. Rules are pluggable and prioritized. The system supports composite-target extraction (e.g., host+port) when appropriate.

5. Implementation
We implemented the module in Node.js with Express for APIs and Mongoose for MongoDB integration. Key implementation points:
- Normalization utilities map diverse event schemas into the canonical model.
- The correlation engine provides functions `buildScanAlertCorrelations(scans, alerts, limit)` which return grouped correlations and an `overview` object with `total` and `targets` fields.
- Injection utilities for demos create sample scans and alerts with recent timestamps and distinct targets to illustrate behavior.

The implementation emphasizes testability: unit tests cover normalization, matching rules, grouping semantics, and API contract tests verify `totalCount` and `overview` are returned for UI consumption.

6. Evaluation
6.1 Experimental Setup
We evaluated the module on two workloads:
- Synthetic workload: controlled sets of scans and alerts with varying degrees of overlap to measure output size and processing latency.
- Replay workload: a trimmed dataset derived from application logs with injected simulated scan traffic.
ff
Metrics: correlation output size, reduction ratio (pairwise vs grouped), API latency, and analyst-relevant precision/recall proxies.

6.2 Results
- Reduction ratio: grouping by `target` reduced correlation counts by up to 95% in high-overlap scenarios compared to pairwise outputs, approximating the distinct-target count.
- Latency: for inputs up to 10k events, correlation building and grouping executed within 200–400 ms on commodity VM instances; latencies scale linearly with event count and are amenable to batching.
- Analyst signal: when measured by the number of distinct investigative targets surfaced per unit time, the grouped approach improved effective throughput and reduced redundant investigation steps.

6.3 Discussion
The results confirm that grouping by semantic targets yields more compact, actionable outputs and scales acceptably for small-to-medium deployments. The performance depends on indexing strategies in the DB and efficient in-memory grouping structures.

7. Case Study: Demo Injection Semantics
During development, we observed that demo injection endpoints that replaced persisted records led to confusing UI observations. We changed the semantics to append demo records and rebuild correlations against current persistence. This demonstrates an operational principle: instrumentation and demo utilities should be non-destructive and reflect production semantics to avoid misleading analytics.

8. Related Work
Entity-centric aggregation has precedent in SIEM designs and research on alert triage [1,3]. Our work differs by focusing on lightweight integrability in full-stack app contexts and on explicit API contracts (e.g., `totalCount`, `overview`) that drive concise UI summaries. Other research in log clustering and event abstraction informs our matching heuristics and grouping strategies [4,5].

9. Limitations and Threats to Validity
- The module relies on the correctness of target extraction. Mis-extraction can lead to under- or over-grouping.
- Heuristic matching introduces tunable thresholds; misconfiguration can cause mergers that hide distinct incidents.
- Evaluation uses synthetic and replayed data; further validation on broad operational datasets is needed.

10. Future Work
- Automated target canonicalization using machine learning to reconcile multiple identifiers (IP, hostname, account) into unified entities.
- Graph-based correlation that preserves target grouping while highlighting cross-target lateral movement.
- Integration with streaming event pipelines (e.g., Kafka) for near-real-time, high-throughput correlation.

11. Conclusion
We presented the Security Module, a practical subsystem for ingesting, alerting, and correlating heterogeneous telemetry within full-stack applications. The key design choice—grouping correlations by semantic targets—reduces output inflation, aligns with analyst workflows, and preserves performance. Our implementation and experiments show that these benefits are achievable with modest engineering effort and can significantly improve the usability of in-app security analytics.
